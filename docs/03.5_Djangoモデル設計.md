# 2. モデル設計方針

本システムでは、以下の方針でモデルを設計する。

- Django ORMを利用する
- SQLiteで運用開始する
- 将来的なPostgreSQL移行を考慮する
- 共通項目はBaseModelに集約する
- 法人情報と病院情報を分離する
- 営業分析は原則としてテーブル化せず、Service層で計算する
- 利用者個人情報は保持しない
- Version0.1では法人サービスはBooleanで管理する
- 将来的には法人サービスを別テーブル化する余地を残す

# 4. BaseModel

主要モデルに共通する項目を定義する抽象モデル。

```python
class BaseModel(models.Model):
    memo = models.TextField(blank=True)
    is_active = models.BooleanField(default=True)
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        abstract = True
```

# 6. Hospital モデル

病院マスタを管理する。

病院は必ずしも法人情報が完全に判明しているとは限らないため、`corporation` は `null=True, blank=True` とする。

```python
class Hospital(BaseModel):
    corporation = models.ForeignKey(
        Corporation,
        on_delete=models.SET_NULL,
        null=True,
        blank=True,
        related_name="hospitals"
    )

    hospital_name = models.CharField(max_length=255)
    address = models.CharField(max_length=255, blank=True)
    phone = models.CharField(max_length=50, blank=True)
    fax = models.CharField(max_length=50, blank=True)
    homepage_url = models.URLField(blank=True)

    total_beds = models.IntegerField(default=0)
    general_beds = models.IntegerField(default=0)
    chronic_beds = models.IntegerField(default=0)
    psychiatric_beds = models.IntegerField(default=0)
    community_beds = models.IntegerField(default=0)
    recovery_beds = models.IntegerField(default=0)
    disability_beds = models.IntegerField(default=0)
    other_beds = models.IntegerField(default=0)

    has_regional_cooperation = models.BooleanField(default=False)
    regional_department_name = models.CharField(max_length=255, blank=True)

    msw_count = models.IntegerField(default=0)
    discharge_nurse_count = models.IntegerField(default=0)

    def __str__(self):
        return self.hospital_name
```

# 10. SalesAnalysisについて

営業分析は、初期段階ではテーブルとして保存しない。

理由は以下の通り。

- 病床数が変わると再計算が必要になる
- 法人サービス情報が変わると再計算が必要になる
- 営業履歴や紹介実績が増えるたびに更新が必要になる
- 計算結果を保存すると、元データとの不整合が起きやすい

そのため、営業分析は `analytics` アプリの Service層で計算する。

```text
Hospital
  +
Corporation
  +
VisitRecord
  +
ReferralRecord
  ↓
AnalyticsService
  ↓
営業スコア
```

# 12. 法人サービス管理の将来拡張

Version0.1では、法人サービスはBooleanで管理する。

将来的に詳細管理が必要になった場合は、法人サービスを別テーブル化する。

```text
Corporation
  1
  │
  多
CorporationService
```

例：

```python
class CorporationService(BaseModel):
    corporation = models.ForeignKey(
        Corporation,
        on_delete=models.CASCADE,
        related_name="services"
    )

    service_type = models.CharField(max_length=100)
    service_name = models.CharField(max_length=255)
    address = models.CharField(max_length=255, blank=True)
    phone = models.CharField(max_length=50, blank=True)
    capacity = models.IntegerField(default=0)
```

ただし、初期段階ではシンプルさを優先し、Boolean管理とする。

# 13. on_delete方針

|関係|on_delete|理由|
|---|---|---|
|Corporation → Hospital|SET_NULL|法人情報が削除されても病院情報は残す|
|Hospital → ContactPerson|CASCADE|病院削除時は担当者も削除対象|
|Hospital → VisitRecord|CASCADE|病院削除時は営業履歴も削除対象|
|Hospital → ReferralRecord|CASCADE|病院削除時は紹介実績も削除対象|
|ContactPerson → VisitRecord|SET_NULL|担当者が退職しても訪問履歴は残す|

# 14. モデル実装時の注意点

- 主要モデルはBaseModelを継承する
- `blank=True` はフォーム入力任意を意味する
- `null=True` はDB上でNULL許可が必要な場合のみ使用する
- 日付が未定の場合は `null=True, blank=True` とする
- 文字列項目は原則 `blank=True` を使用し、NULLは使わない
- 数値項目は `default=0` を設定する
- 外部キーには `related_name` を設定する
- 営業分析はテーブル保存せず、Service層で計算する
- 法人サービス詳細管理はVersion1.0以降で検討する
